````markdown
## 1. 项目简介

`nail-ai-ops` 是一个面向单店美甲商家的 AI 智能运营与客户端试戴展示系统。

项目围绕“美甲 AI 试戴 + 商家智能运营”展开，试图解决美甲消费场景中的两个核心问题：

第一，用户在浏览美甲款式时，往往只能看到款式图，却很难判断该款式是否适合自己的手型、肤色、日常穿搭和使用场景，导致选择成本较高。

第二，美甲商家虽然拥有大量款式图和一定运营经验，但在款式主推、上新推广、活动文案生成、客户咨询转化等环节仍然依赖人工经验，缺少轻量化的 AI 运营辅助工具。

因此，本项目设计了一个最小可行系统：

> 商家端负责上传或选择款式库，并通过 AI 生成主推策略和宣传文案；  
> 客户端负责承接用户需求，让用户选择款式、上传手图或使用示例手图，并生成美甲试戴效果图。

项目目标不是做一个完整的大型美团平台，也不是做复杂电商系统，而是在 Hackathon 时间内完成一个可演示、可解释、可迭代的 AI 原型。

## 2. 项目背景

在美甲消费场景中，用户通常会经历以下决策过程：

```text
看到美甲图
→ 觉得款式好看
→ 犹豫是否适合自己
→ 咨询商家
→ 反复对比
→ 决定是否预约
````

这个过程中存在明显的决策阻碍：

```text
款式图好看，但不知道自己上手效果如何
不知道这个颜色是否显白
不知道款式是否适合上课、上班、约会、拍照等场景
需要反复咨询商家，沟通成本较高
如果没有直观效果图，用户容易放弃预约
```

对于商家来说，也存在运营压力：

```text
店铺款式数量多，但不知道该主推哪几款
新款上架后缺少快速推广方式
宣传文案需要手动编写
店员推荐高度依赖个人经验
用户咨询过程重复，转化效率不稳定
```

因此，美甲场景天然适合引入 AI：

```text
AI 可以理解用户自然语言需求
AI 可以根据标签体系推荐款式
AI 可以生成试戴效果图
AI 可以帮助商家生成运营策略
AI 可以生成美团页面、小红书、朋友圈等宣传文案
```

## 3. 项目定位

本项目定位为：

> 单店美甲 AI 智能运营助手 + 客户端试戴展示页

项目不做大而全平台，不做复杂商业闭环，而是聚焦一个清晰场景：

```text
一家美甲店
一批美甲款式
一个商家运营目标
一个用户试戴流程
一条从商家运营到用户转化的 AI 辅助链路
```

项目核心闭环如下：

```text
商家款式库
→ AI 理解款式风格
→ AI 生成主推策略和宣传文案
→ 客户端展示推荐款式
→ 用户描述需求并选择款式
→ 上传手图 / 使用示例手图
→ AI 生成试戴效果图
→ 用户咨询 / 预约 / 换一款
```

## 4. 核心目标

本项目有三个核心目标。

### 4.1 降低用户选款成本

用户不需要只靠静态款式图想象效果，而是可以通过上传手图或使用示例手图快速查看试戴结果。

系统希望帮助用户回答这些问题：

```text
这款美甲适合我吗？
这个颜色会不会显黑？
这个风格适合上课 / 上班 / 约会吗？
我想要温柔、显白、日常一点的款式，应该选哪几款？
```

### 4.2 提高商家运营效率

商家不需要完全依赖人工经验判断主推款式，可以通过 AI 结合款式标签和运营目标生成策略。

系统希望帮助商家回答这些问题：

```text
本周应该主推哪些款式？
新款应该怎么介绍？
适合学生用户的款式有哪些？
适合上班族的款式有哪些？
美团页面上的宣传文案怎么写？
```

### 4.3 构建可演示的 AI 业务闭环

Hackathon 项目不只是展示一个孤立的 AI 生成能力，而是要体现 AI 如何嵌入业务流程。

本项目希望展示：

```text
AI 不是只负责聊天
AI 可以参与推荐
AI 可以参与图像试戴
AI 可以参与商家运营
AI 可以连接 C 端用户体验和 B 端商家经营
```

## 6. 用户端流程

用户端，也就是客户端，面向普通消费者。

用户端目标是完成：

```text
需求描述
→ 款式推荐
→ 选择款式
→ 上传手图
→ 生成试戴效果
→ 咨询或预约
```

完整流程如下：

```text
用户进入首页
→ 输入需求描述
→ 通过自然语言 / 类别筛选表达偏好
→ 系统读取店铺款式库
→ AI 或规则系统生成推荐款式列表
→ 用户选择一款美甲
→ 用户上传手图
→ 如果未上传，则使用示例手图
→ 系统组合“用户选择的美甲款式 + 用户手图 / 示例手图”
→ AI 生成试戴结果图
→ 用户查看效果
→ 用户点击咨询 / 预约 / 换一款
```

### 6.1 用户需求输入示例

```text
我想要一个适合上班、不太夸张、显白一点的美甲。
```

```text
想要温柔一点，适合日常和约会，颜色不要太深。
```

```text
有没有适合学生党、显手白、价格看起来不要太贵的款式？
```

注意：当前 MVP 暂不做真实价格推荐。这里的“不要太贵”只作为风格倾向理解，例如推荐简约、日常、低复杂度款式。

### 6.2 用户端页面设计

用户端可以拆成以下页面或区域：

```text
首页 HomePage
需求输入 RecommendPage
款式推荐列表 NailList
款式详情 NailDetail
上传手图 UploadHandImage
试戴结果 TryOnResult
咨询 / 预约 / 换一款操作区 ActionPanel
```

```text
→ 选择运营目标
→ AI 生成策略

```

完整流程如下：

```text
商家上传或选择款式库
→ 商家选择运营目标
→ AI 理解款式风格和标签
→ AI 生成主推策略
→ AI 生成宣传文案
→ 输出客户端展示内容
```

### 7.1 商家运营目标示例

```text
提升新款曝光
提升预约转化
推广节日款式
主推通勤日常款
吸引学生用户
吸引上班族用户
打造本周主推款
```

### 7.2 商家端输出内容示例

商家选择目标：

```text
提升新款曝光
```

AI 输出主推策略：

```text
建议本周主推低饱和裸粉法式、细闪猫眼和简约腮红甲。
低饱和裸粉法式适合日常通勤用户，决策成本较低；
细闪猫眼视觉效果更强，适合用于首页展示和社交平台种草；
简约腮红甲适合学生用户和年轻客群，可作为低门槛推荐款。
```

AI 输出宣传文案：

```text
低饱和裸粉法式，温柔又不夸张。
适合上班、上课和日常约会，显手白，也更容易搭配日常穿搭。
```

## 9. AI 能力设计

本项目中的 AI 能力主要分为四类。

### 9.1 用户需求理解

用户输入自然语言后，AI 提取用户偏好。

示例输入：

```text
我想要一个温柔、显白、适合上课的美甲。
```

AI 需要理解：

```text
风格偏好：温柔
颜色需求：显白
使用场景：上课 / 日常
排除条件：不要太夸张
```

输出结构可以是：

```json
{
  "style_preferences": ["温柔", "简约"],
  "color_preferences": ["显白", "低饱和"],
  "scene_preferences": ["上课", "日常"],
  "avoid": ["夸张", "过度华丽"]
}
```

### 9.2 款式推荐

系统根据用户需求和款式标签进行匹配。

推荐依据包括：

```text
风格标签
颜色标签
工艺标签
场景标签
人群标签
款式描述
```

推荐输出包括：

```text
款式名称
款式图片
标签
推荐理由
试戴按钮
```

### 9.3 美甲试戴生成

用户选择款式并上传手图后，系统调用图像生成或图像编辑能力，生成试戴结果图。

如果用户没有上传手图，则使用系统内置示例手图完成流程演示。

当前 MVP 的重点是跑通流程，而不是追求工业级真实 AR 效果。

### 9.4 商家运营策略生成

商家选择运营目标后，AI 根据款式库和标签体系生成运营建议。

示例输入：

```text
运营目标：提升新款曝光

款式库概况：
- 温柔风款式较多
- 裸粉、低饱和、猫眼款占比较高
- 新款中包含法式、腮红甲、细闪猫眼等类型
```

示例输出：

```text
本周建议主推低饱和裸粉法式、细闪猫眼和腮红甲。
其中低饱和裸粉法式适合日常通勤场景，可以作为低决策成本款式；
细闪猫眼视觉冲击更强，适合放在首页 Banner；
腮红甲适合学生用户和年轻女性用户，可以配合“开学季”“日常显白”等主题进行推广。
```

## 11. 推荐逻辑设计

当前项目不做复杂推荐模型，采用“自然语言理解 + 标签匹配 + 规则排序”的轻量推荐方案。

### 11.1 推荐输入

```text
用户自然语言需求
用户选择的风格标签
店铺款式库
商家当前运营目标
```

### 11.2 推荐过程

```text
1. 解析用户需求
2. 提取关键词和偏好标签
3. 与款式库中的 style_tags / color_tags / scene_tags 等进行匹配
4. 根据匹配数量计算基础分
5. 根据商家运营目标进行轻微加权
6. 输出推荐款式列表
```

### 11.3 推荐输出

```json
{
  "nail_id": "nail_001",
  "name": "奶油裸粉法式",
  "score": 0.92,
  "reason": "这款美甲符合你提出的“温柔、显白、适合日常”的需求，颜色低饱和，不夸张，适合上课和通勤。"
}
```

### 11.4 为什么不做复杂模型

当前阶段不做复杂推荐模型，原因是：

```text
Hackathon 时间有限
真实用户行为数据暂不可用
美甲款式库规模较小
标签匹配方案更可解释
更适合 Demo 展示和答辩说明
```

## 14. Demo 展示路径

推荐最终展示时按以下路径演示：

```text
1. 商家进入后台
2. 商家选择或上传店铺款式库
3. 商家选择运营目标，例如“提升新款曝光”
4. AI 生成主推策略
5. AI 生成宣传文案
6. 客户端展示对应推荐款式
7. 用户进入客户端首页
8. 用户输入需求，例如“想要温柔显白、适合上课的款式”
9. 系统推荐相关美甲款式
10. 用户选择一款美甲
11. 用户上传手图或使用示例手图
12. AI 生成试戴效果图
13. 用户查看试戴效果
14. 用户点击咨询 / 预约 / 换一款
```

这条路径可以体现完整闭环：

```text
商家运营目标
→ AI 生成策略
→ 客户端展示
→ 用户需求输入
→ AI 推荐
→ AI 试戴
→ 用户转化动作
```
